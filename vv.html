1. Матрица трассируемости

Матрица связывает требования (SR, SYS, NFR) с методом проверки и конкретным артефактом/тест-кейсом.

Методы проверки:

  • Тест — функциональный или интеграционный тест-кейс
  • Нагрузочный тест — тест производительности (k6 / JMeter)
  • Инспекция — проверка кода, конфигурации или документации
  • Анализ — расчёт или моделирование без запуска системы
  • Демо — демонстрация работающей системы

2. Тест-кейсы

ТК-01. Запуск экзамена с прокторингом в реальном времени (E2E happy path)

ПолеЗначение
IDTC-001
Связанные требованияSR-01, SR-02, SYS-01, SYS-14, SYS-15, SYS-16
Предусловия1. Студент авторизован через SSO. 2. Экзамен опубликован с режимом realtime. 3. Текущее время — в пределах окна. 4. Медиасервер доступен. 5. Проктор назначен и авторизован.
Шаги1. Студент открывает страницу экзамена. 2. Читает политику прокторинга, нажимает «Согласен». 3. Система выполняет camera check. 4. Студент нажимает «Начать экзамен». 5. LMS+P устанавливает WebRTC-соединение с медиасервером. 6. Студент видит вопросы; в углу — «Запись ведётся». 7. Проктор видит видеопоток студента в портале.
Ожидаемый результат1. Политика прокторинга отображена. 2. WebRTC-соединение установлено ≤30 сек. 3. Видеопоток у проктора. 4. В БД: запись сессии status=active.
Пост-условияproctoring_sessions: status=active, start_time заполнен.
ОкружениеТестовый стенд, тестовые аккаунты, mock SSO.

ТК-03. Сохранение ответов при разрыве соединения

ПолеЗначение
IDTC-003
Связанные требованияSYS-09, NFR-08
ПредусловияСтудент в активной сессии, ответил на 5 вопросов из 10.
Шаги1. Студент переходит к вопросу №6. 2. Симуляция разрыва (DevTools → Offline). 3. Студент вводит ответ на вопрос №6. 4. Соединение восстанавливается. 5. Студент обновляет страницу.
Ожидаемый результат1. Ответы 1–5 сохранены на сервере. 2. Ответ №6 из localStorage синхронизирован при восстановлении. 3. Прогресс «5/10» или «6/10» отображён корректно.
Пост-условияТаблица answers: 5 или 6 записей для данной сессии.
ОкружениеChrome DevTools, тестовый стенд.

ТК-04. Детекция тревоги — смена браузерной вкладки

ПолеЗначение
IDTC-004
Связанные требованияSYS-17, SYS-18, SYS-23, SR-12
ПредусловияСтудент в активной сессии realtime. Проктор авторизован и наблюдает.
Шаги1. Студент переключается на другую вкладку и остаётся там 4 секунды. 2. Студент возвращается на страницу экзамена.
Ожидаемый результат1. Проктор получает push-уведомление о tab_switch в течение 5 сек. 2. В proctoring_events: запись с event_type=tab_switch, confidence_score, duration_ms≈4000. 3. Студенту NOT показывается сообщение о тревоге.
Пост-условияСобытие сохранено в БД; студент не уведомлён.
ОкружениеДва браузера (студент + проктор), тестовый стенд.

ТК-05. Вынесение вердикта проктором

ПолеЗначение
IDTC-005
Связанные требованияSYS-19, SYS-20, SR-06
ПредусловияСессия завершена. Запись в S3. 2 события: tab_switch и multiple_faces.
Шаги1. Проктор открывает сессию. 2. Нажимает на tab_switch → «Не нарушение». 3. Нажимает на multiple_faces, просматривает скриншот → «Нарушение», тип: «Посторонний», комментарий. 4. «Завершить проверку» → итог: «Не зачтено».
Ожидаемый результат1. Оба события получили вердикт с proctor_id и verdict_timestamp. 2. Итоговый статус сессии: rejected. 3. В Assessment Service — событие proctoring.verdict.submitted. 4. Преподаватель уведомлён.
Пост-условияРезультат помечен «Аннулирован (нарушение прокторинга)».
ОкружениеТестовый стенд, предзаполненные данные с mock-событиями.

ТК-06. Переключение в асинхронный режим при отказе медиасервера

ПолеЗначение
IDTC-006
Связанные требованияSYS-15, NFR-08, SR-11
ПредусловияМедиасервер в офлайн (симуляция). Студент начинает экзамен с режимом realtime.
Шаги1. Студент нажимает «Начать экзамен». 2. Система пытается WebRTC — таймаут 10 сек. 3. Студент видит сообщение и выбирает «Продолжить».
Ожидаемый результат1. Через 10 сек: «Прямое наблюдение недоступно. Экзамен будет записан и проверен после». 2. Сессия запускается в async режиме. 3. API: HTTP 503 с suggestion: async_mode.
Пост-условияСессия создана с proctoring_mode=async_fallback.
ОкружениеТестовый стенд с отключённым медиасервером.

ТК-08. Передача оценок в SIS с retry при сбое

ПолеЗначение
IDTC-008
Связанные требованияSR-07, SYS-24, SYS-25
ПредусловияПреподаватель подтвердил оценки (3 студента). SIS-мок: первые 2 вызова → 503, третий → 200.
Шаги1. Преподаватель нажимает «Передать в деканат». 2. 1-я попытка → 503. 3. Retry через 1 мин → 503. 4. Retry через 5 мин → 200.
Ожидаемый результат1. После 3-й попытки оценки переданы. 2. Статус UI: «Передано в деканат». 3. В аудит-логе: 3 записи с кодами 503, 503, 200. 4. Студенты получили email.
Пост-условияgrade_sync_log: 3 записи для данного курса.
ОкружениеSIS-мок (возвращает 503 первые 2 раза).

ТК-09. Нагрузочный тест: 500 одновременных сессий

ПолеЗначение
IDTC-009
Связанные требованияNFR-02, NFR-03, NFR-06, SR-10
Предусловия2 реплики каждого сервиса. 500 тестовых аккаунтов. Медиасервер с 4 воркерами.
Шаги1. k6 запускает 500 сессий за 5 мин (ramp-up). 2. Поддерживает 500 активных сессий 30 мин. 3. Каждая сессия отправляет ответы каждые 30 сек. 4. Фиксируются: latency p95/p99, error rate, время WebRTC.
Ожидаемый результат1. API p95 latency ≤ 500 мс. 2. Error rate ≤ 1%. 3. WebRTC установлен у ≥97% сессий ≤30 сек. 4. CPU медиасервера ≤ 80%.
Пост-условияВсе 500 сессий завершились корректно; ни один ответ не потерян.
ОкружениеНагрузочный стенд, k6, Grafana.

ТК-12. Доступ к видеозаписи, аудит-лог и TTL URL

ТК-13. Полный цикл создания курса и получения результатов

ПолеЗначение
IDTC-013
Связанные требованияSR-03, SR-04, SR-13, SYS-05, SYS-06, SYS-07, SYS-08, SYS-12, SYS-26, SYS-27
ПредусловияПреподаватель авторизован. SIS-мок доступен и возвращает 1 студента.
Шаги1. Преподаватель создаёт курс с 2 модулями и тестом (3 вопроса: 2 закрытых + 1 открытый). 2. Загружает PDF-материал (<2 ГБ). 3. Публикует курс. 4. Студент отвечает на тест и завершает его. 5. Преподаватель проверяет открытый вопрос. 6. Преподаватель открывает отчёт успеваемости и экспортирует CSV.
Ожидаемый результат1. Курс создан с иерархией ≥3 уровня. 2. PDF загружен. 3. Студент зачислен автоматически из SIS. 4. Закрытые вопросы оценены автоматически; открытый — в очереди. 5. После ручной проверки — статус «Проверено», итоговый балл рассчитан. 6. Отчёт содержит: средний балл, распределение, прогресс. CSV скачан ≤10 сек.
Пост-условияВ БД: курс, тест, 3 ответа, итоговый балл, запись в grade_sync_queue.
ОкружениеТестовый стенд, SIS-мок.
