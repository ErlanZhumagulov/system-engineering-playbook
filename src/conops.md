## 1. Роли участников

| Роль | Определение | Основные действия | Привилегии в LMS+P |
|---|---|---|---|
| **Студент** | Обучающийся, зачисленный на курс | Просматривает курсы, проходит тесты и экзамены, получает оценки | Доступ только к своим курсам и результатам |
| **Преподаватель** | Автор курса, педагог | Создаёт курсы и тесты, проверяет работы, просматривает успеваемость группы, управляет правилами прокторинга | Полный доступ к своим курсам, чтение статистики по своим студентам |
| **Проктор** | Наблюдатель на экзамене | Мониторит видеосессии, просматривает записи, выносит вердикты по нарушениям | Доступ к видеосессиям и событиям тревоги назначенных экзаменов |
| **Администратор** | Сотрудник IT-отдела / деканата | Управляет пользователями, ролями, интеграциями, системными настройками, логами | Полный доступ к настройкам, без доступа к содержанию экзаменов |

## 2. Сценарии использования

### Сценарий 1: Экзамен с прокторингом в реальном времени

**Предусловие:** Преподаватель создал экзамен с режимом «Реальное время». Проктор назначен на сессию. Студент зачислен на курс и получил уведомление.

**Основной поток:**

1. Студент входит в систему через SSO за 15 минут до начала
2. Открывает страницу экзамена — система проверяет расписание, доступ разрешён
3. Студент читает политику прокторинга и нажимает «Согласен»; система выполняет camera check
4. Студент нажимает «Начать экзамен» — Proctoring Service запрашивает WebRTC-соединение с медиасервером
5. Соединение установлено (≤30 сек); в углу экрана — индикатор «Запись ведётся»
6. Проктор видит видеопоток студента в портале проктора
7. Студент отвечает на вопросы; Assessment Service фиксирует ответы с временными метками
8. Proctoring Service детектирует события: уход из кадра, смена вкладки, посторонний звук
9. При confidence_score > 0.8 — push-уведомление проктору в течение 5 секунд
10. По истечении времени система блокирует возможность ответа и автоматически завершает сессию
11. Запись сохраняется в S3; Assessment Service рассчитывает результат по автоматическим вопросам

@drawio{https://github.com/ErlanZhumagulov/system-engineering-playbook/blob/main/src/diagrams/seq-exam-realtime.drawio}

**Исключение — разрыв видеосоединения:**

1. Медиасервер фиксирует разрыв дольше 30 секунд
2. LMS+P приостанавливает таймер экзамена и уведомляет студента и проктора
3. Студент получает инструкцию перезагрузить страницу
4. При восстановлении соединения (<5 минут) таймер возобновляется, сессия продолжается
5. Если соединение не восстановлено — проктор принимает решение: продлить сессию или зафиксировать как технический сбой

**Режим деградации — медиасервер полностью недоступен:**

1. LMS+P не может установить WebRTC-соединение (таймаут 10 сек)
2. Система автоматически предлагает студенту переключиться в режим «Асинхронный»
3. Студент получает уведомление: «Прямой эфир недоступен, используется запись на устройстве»
4. По завершении экзамена браузерная запись загружается в S3 для последующего просмотра проктором
5. Проктор уведомляется о переключении режима; сессия помечается `proctoring_mode=async_fallback`

### Сценарий 2: Создание курса и зачисление студентов

**Предусловие:** Преподаватель авторизован. Курс ещё не создан. SIS содержит актуальные данные о зачислении.

**Основной поток:**

1. Преподаватель открывает редактор курса, создаёт модули и загружает материалы (PDF, видео)
2. Добавляет тест: создаёт вопросы (одиночный выбор, множественный, открытый текст)
3. Задаёт настройки теста: время, количество попыток, режим прокторинга
4. Нажимает «Опубликовать» — LMS+P запрашивает из SIS список зачисленных студентов
5. Студентам предоставляется доступ к курсу; они получают email-уведомление
6. Преподаватель просматривает прогресс студентов в разделе успеваемости в режиме реального времени

**Исключение — ошибка загрузки файла:**

1. Преподаватель загружает файл >2 ГБ или неподдерживаемого формата
2. Система отображает ошибку: «Максимальный размер файла — 2 ГБ. Поддерживаемые форматы: PDF, MP4, DOCX, PPTX, PNG, JPEG»
3. Курс сохраняется в черновике; материал не загружается
4. Преподаватель конвертирует файл и повторяет загрузку

**Режим деградации — SIS недоступен при публикации:**

1. LMS+P не может получить список зачисленных студентов из SIS
2. Система публикует курс, но откладывает автоматическое зачисление в очередь
3. При восстановлении SIS — зачисление выполняется автоматически
4. Администратор получает уведомление о задержке зачисления с указанием курса и времени ожидания

### Сценарий 3: Асинхронный прокторинг — проверка записи

**Предусловие:** Студент прошёл экзамен. Запись загружена в S3. Проктор назначен на проверку записей данного экзамена.

**Основной поток:**

1. Проктор открывает очередь записей в портале
2. Система загружает запись с pre-signed URL из S3 (TTL = 4 часа)
3. Проктор просматривает видео; система отображает временную шкалу с метками тревог
4. Проктор нажимает на метку тревоги — видео перематывается к событию
5. Проктор выносит вердикт по каждому событию: «Нарушение» / «Не нарушение» / «Требует эскалации»
6. При выборе «Нарушение» — открывается форма с полем комментария и выбором типа нарушения
7. Итоговый вердикт (допустить / не допустить к результатам) сохраняется с `proctor_id` и `timestamp`
8. Преподаватель получает уведомление о вердикте; система обновляет статус результата

**Исключение — запись недоступна в S3:**

1. При попытке открыть сессию система обнаруживает отсутствие объекта в S3
2. Проктор видит статус «Запись недоступна — обратитесь к администратору»
3. Администратор получает автоматический алерт с `session_id` и `exam_id`
4. Администратор выясняет причину: сбой загрузки после экзамена или технический сбой на стороне студента
5. Принимается решение: провести повторный экзамен или засчитать результат на основе автодетекции событий

**Режим деградации — S3 недоступен во время проверки:**

1. Проктор пытается открыть запись — S3 возвращает ошибку (timeout или 5xx)
2. Система отображает сообщение: «Хранилище временно недоступно. Повторите через несколько минут»
3. Событие тревоги и метаданные сессии остаются доступны (хранятся в PostgreSQL)
4. Проктор может изучить список автодетектированных событий без видео и отложить вынесение вердикта
5. При восстановлении S3 — проктор возвращается к записи с сохранённым прогрессом просмотра

### Сценарий 4: Ручная проверка открытых вопросов

**Предусловие:** Студент завершил тест с открытыми вопросами. Assessment Service поставил задачи ручной проверки в очередь.

**Основной поток:**

1. Преподаватель получает email-уведомление о задачах в очереди
2. Открывает очередь — видит список ответов (имя студента скрыто, если включена слепая проверка)
3. Читает ответ, вводит балл в допустимом диапазоне, добавляет комментарий
4. Сохраняет оценку — Assessment Service пересчитывает итоговый балл
5. Если все вопросы проверены — статус теста меняется на «Проверено»
6. Студент получает уведомление и видит итоговый балл с комментарием преподавателя

**Исключение — преподаватель вводит балл вне допустимого диапазона:**

1. Преподаватель вводит балл выше максимального (например, 110 из 100)
2. Система отображает inline-ошибку: «Балл должен быть от 0 до 100»
3. Форма блокирует сохранение до исправления значения
4. Преподаватель исправляет балл и сохраняет

**Режим деградации — преподаватель не проверяет задачи >N дней:**

1. Через 3 дня бездействия система отправляет напоминание преподавателю
2. Если через 7 дней очередь не обработана — уведомление администратору с указанием курса и числа непроверенных работ
3. Администратор может переназначить задачи другому преподавателю
4. Студент, ожидающий результатов >7 дней, получает уведомление: «Проверка задерживается, ожидайте»

### Сценарий 5: Передача итоговых оценок в деканат

**Предусловие:** Все экзамены курса проведены. Преподаватель проверил открытые вопросы. SIS доступен.

**Основной поток:**

1. Преподаватель открывает раздел «Итоговые оценки» курса
2. Проверяет список: все студенты имеют статус «Оценка выставлена»
3. Нажимает «Передать в деканат»
4. SIS Sync Service отправляет POST-запрос в SIS с массивом оценок: `student_id`, `course_id`, `grade`, `grade_date`, `status`
5. SIS возвращает подтверждение; LMS+P фиксирует статус «Передано в деканат» с timestamp
6. Студенты получают email-уведомление о передаче оценок

**Исключение — ошибка валидации в SIS:**

1. SIS возвращает ошибку: `student_id` не найден или `course_id` не соответствует учебному плану
2. LMS+P отображает список проблемных записей с кодом ошибки от SIS
3. Администратор вручную сверяет данные и исправляет расхождение
4. Преподаватель повторяет передачу только для проблемных записей

**Режим деградации — SIS недоступен:**

1. SIS Sync Service фиксирует HTTP 5xx или timeout — помещает оценки в очередь со статусом «Ожидает передачи»
2. Система автоматически повторяет попытку с интервалами: 1, 5, 15, 60 минут
3. Администратор видит в дашборде: «Очередь передачи: N записей, следующая попытка через X мин»
4. После 5 неудачных попыток — алерт администратору по email
5. При восстановлении SIS — передача выполняется автоматически, статус обновляется; студенты уведомляются
