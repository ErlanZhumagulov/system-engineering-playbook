## 1. Суть кейса и миссия

**Система:** LMS+P — платформа дистанционного обучения с интегрированным прокторингом.

**Цель системы:** Обеспечить образовательным организациям надёжный, доступный и справедливый инструмент дистанционного обучения и верифицированной оценки знаний — с защитой от академической нечестности, соблюдением требований приватности и минимальной нагрузкой на административный персонал.

**Ключевые задачи LMS+P:**

- Размещение курсов, лекций, заданий и учебных материалов
- Проведение тестов и экзаменов с автоматической проверкой
- Видеопрокторинг в реальном времени и в асинхронном режиме
- Автодетекция нарушений (смена вкладок, посторонние лица, посторонний звук)
- Передача итоговых оценок в систему деканата (SIS)
- Отчётность об успеваемости для студентов, преподавателей и администрации
- Единый вход через SSO организации

## 2. Контекст — внешние сущности и потоки

LMS+P функционирует в экосистеме образовательной организации. Система выступает как связующий слой между IdP/SSO, системой деканата (SIS), медиасервером, объектным хранилищем и почтовым сервером — объединяя процессы обучения, оценивания, прокторинга и академической отчётности в единый управляемый контур.

### Таблица внешних сущностей и потоков

| Внешняя сущность | Тип | Входящие в LMS+P | Исходящие из LMS+P |
|---|---|---|---|
| **Студент** | Пользователь | Ответы на задания, видеопоток (камера/экран), запуск/завершение сессии | Материалы курсов, результаты тестов, оценки, уведомления о расписании |
| **Преподаватель** | Исполнитель | Учебные материалы, вопросы для тестов, настройки прокторинга, оценки за эссе | Отчёты успеваемости, события нарушений, список задач ручной проверки |
| **Проктор** | Исполнитель | Вердикт по нарушению (подтверждено / снято) | Видеопоток сессии, события тревоги, записи для асинхронного просмотра |
| **Администратор** | Исполнитель | Настройки пользователей, курсов, интеграций | Системные логи, отчёты, статусы интеграций |
| **IdP / SSO** (Keycloak, Azure AD) | Система аутентификации | Токен + атрибуты пользователя, роли из LDAP/AD | Redirect-запрос аутентификации |
| **WebRTC Медиасервер** (Janus / MediaSoup) | Стриминговая платформа | Видео/аудио поток студента (via WebRTC) | Сигналы инициализации соединения, команды записи |
| **S3-хранилище** | Объектное хранилище | URL-ссылка на загруженную запись | Видеозапись экзаменационной сессии |
| **SIS / Деканат** | Учётная система | Данные о зачислении, учебные планы, расписание экзаменов | Итоговые оценки, статусы прохождения экзаменов |
| **SMTP / Почтовый сервер** | Канал связи | Статус доставки, bounce-уведомления | Письма студентам, преподавателям, проктарам |
| **Античит-агент** (опционально) | Агент на устройстве | Статус lockdown-браузера (активен / нарушение) | Команда активации / деактивации |
| **APM / Мониторинг** (Prometheus/Grafana) | Наблюдаемость | Алерты о превышении порогов | Метрики, трейсы, логи |
| **BI / Аналитика** (опционально) | Аналитическая система | — | Агрегированные данные успеваемости (ETL-выгрузка) |

### 2.1 Пользователи

Пользователи взаимодействуют с системой через веб-портал или мобильное приложение.

#### Студент

| Действие | Описание | Способ взаимодействия |
|---|---|---|
| Вход в систему | Авторизуется через SSO организации | Редирект на IdP, возврат токена |
| Просмотр курса | Просматривает лекции, материалы, задания | Веб-портал студента |
| Прохождение теста | Отвечает на вопросы в отведённое время | Тестовый модуль LMS+P |
| Запуск экзамена | Инициирует сессию, разрешает доступ к камере и экрану | Модуль прокторинга |
| Завершение экзамена | Отправляет ответы, система закрывает сессию и сохраняет запись | Автоматически или вручную |
| Просмотр результатов | Видит баллы, обратную связь от преподавателя | Личный кабинет |

**Поведенческий профиль:** Студент ожидает стабильной работы видеосвязи и справедливой оценки. Технические сбои во время экзамена вызывают стресс и жалобы. Задержка результатов снижает удовлетворённость.

#### Преподаватель

| Действие | Описание | Способ взаимодействия |
|---|---|---|
| Создание курса | Загружает материалы, структурирует модули | Редактор курсов |
| Создание теста | Формирует вопросы, задаёт настройки прокторинга | Конструктор тестов |
| Просмотр успеваемости | Смотрит сводку по группе, отдельным студентам | Раздел отчётности |
| Ручная проверка | Оценивает эссе и открытые вопросы | Очередь задач проверки |
| Управление нарушениями | Получает уведомление о тревогах, принимает решение | Журнал событий |

**Поведенческий профиль:** Преподаватель ожидает минимального времени на административные задачи. Ручная передача оценок в деканат недопустима. Отсутствие отчётности в реальном времени снижает управляемость группой.

### 2.2 Исполнители

Исполнители — сотрудники образовательной организации, использующие систему как инструмент управления учебным процессом.

| Роль | Основные действия | Частота | Требования к интерфейсу |
|---|---|---|---|
| **Преподаватель** | Создание курсов и тестов, просмотр успеваемости, ручная проверка, реакция на нарушения | Ежедневно | Полнофункциональный редактор, отчёты |
| **Проктор** | Мониторинг видеосессий в реальном времени, просмотр записей, вынесение вердиктов | В период экзаменов | Минималистичный, быстрый, плитки сессий |
| **Администратор** | Управление учётными записями, ролями, интеграциями, системными настройками | По мере необходимости | Панель управления, аудит-логи, статусы |

**Поведенческий профиль исполнителей:**

- **Проктор** должен успевать наблюдать за ≥20 сессиями одновременно — интерфейс обязан быть минималистичным, тревоги должны выводиться автоматически на первый план без ручного поиска.
- **Администратор** не должен вручную переносить данные между LMS и деканатом — любая синхронизация должна выполняться системой автоматически с уведомлением об ошибках.

### 2.3 Интеграции с внешними системами

#### Интеграция №1: IdP / SSO (OIDC / SAML 2.0)

| Параметр | Описание |
|---|---|
| **Тип** | Федеративная аутентификация |
| **Протокол** | OIDC (OpenID Connect) или SAML 2.0 |
| **Данные от IdP → LMS+P** | `sub` (user_id), `email`, `name`, `roles`, `groups` |
| **Данные от LMS+P → IdP** | Authorization Request, redirect_uri |
| **Поведение при недоступности** | Отображает страницу ошибки «Сервис авторизации недоступен», активные сессии (JWT) не прерываются до истечения TTL 8 ч |
| **Настройка** | Администратор указывает Issuer URL, Client ID и Secret в настройках интеграции |

**Обоснование:** LMS+P не хранит пароли и не проводит первичную аутентификацию. Использование корпоративного IdP позволяет применять уже существующую политику паролей и MFA организации, сокращая поверхность атаки и затраты на управление учётными записями. OIDC предпочтителен перед SAML как более современный и гибкий протокол; SAML поддерживается для интеграции с legacy-системами.

---

#### Интеграция №2: WebRTC Медиасервер (Janus / MediaSoup)

| Параметр | Описание |
|---|---|
| **Тип** | Видеостриминг реального времени (SFU — Selective Forwarding Unit) |
| **Протокол** | WebRTC (DTLS-SRTP) + Signaling через WebSocket |
| **Данные от браузера студента → Медиасервер** | SDP-оффер, ICE-кандидаты, видео/аудио медиапоток |
| **Данные от Медиасервера → LMS+P** | Статус подключения, ID потока, событие о разрыве |
| **Данные от Медиасервера → Проктор** | Видеопоток в реальном времени |
| **Поведение при недоступности** | Система переключается в асинхронный режим — студент продолжает экзамен, видео записывается локально |

**Обоснование:** Прямое P2P WebRTC-соединение между студентом и проктором не масштабируется (один проктор не может наблюдать за 20+ студентами через P2P) и не позволяет сохранять запись на сервере. Медиасервер типа SFU решает обе проблемы: проктор подписывается на нужные потоки без нагрузки на устройство студента, а запись формируется централизованно.

---

#### Интеграция №3: SIS / Деканат

| Параметр | Описание |
|---|---|
| **Тип** | REST API двусторонний |
| **Протокол** | HTTPS, JSON |
| **Данные от SIS → LMS+P** | Список зачисленных студентов, учебные планы, расписание экзаменов |
| **Данные от LMS+P → SIS** | Итоговые оценки: `student_id`, `course_id`, `grade`, `date`, `status` |
| **Синхронизация** | Автоматически при выставлении итоговой оценки; ручной запуск доступен администратору |
| **Поведение при недоступности** | LMS+P сохраняет оценки в очередь и отправляет при восстановлении (retry: 1/5/15/60 мин) |

**Обоснование:** Ручной перенос оценок между LMS и деканатом является источником ошибок и задержек. REST API обеспечивает автоматическую двустороннюю синхронизацию: LMS+P получает актуальный список студентов из SIS при публикации курса, а оценки передаются обратно без участия администратора. Очередь с retry исключает потерю данных при временной недоступности SIS.

---

#### Интеграция №4: S3-хранилище

| Параметр | Описание |
|---|---|
| **Тип** | Объектное хранилище |
| **Протокол** | S3 API (совместимый: AWS S3, MinIO, Yandex Object Storage) |
| **Данные от LMS+P → S3** | Видеозапись сессии (MP4/WebM), метаданные (`student_id`, `exam_id`, `timestamp`) |
| **Данные от S3 → LMS+P** | Pre-signed URL для просмотра записи проктором |
| **Срок хранения** | Настраивается администратором (по умолчанию 90 дней, S3 lifecycle policy) |
| **Шифрование** | AES-256 на стороне S3 (SSE-S3 или SSE-KMS) |

**Обоснование:** Видеозапись одной экзаменационной сессии составляет 1–2 ГБ. Хранение в реляционной СУБД невозможно, в файловой системе сервиса — не масштабируется и усложняет резервное копирование. S3-совместимое хранилище обеспечивает горизонтальную масштабируемость, встроенное шифрование, автоматическое управление жизненным циклом объектов (TTL-удаление) и возможность выдачи временных pre-signed URL — без раскрытия прямых ссылок на файлы.

## 3. Границы системы SoI

### 3.1 Внутри SoI

| Компонент / Сервис | Обязанность |
|---|---|
| **Портал студента** | UI для просмотра курсов, прохождения тестов и экзаменов, просмотра оценок |
| **Портал преподавателя** | UI для создания курсов, тестов, просмотра успеваемости, ручной проверки |
| **Портал проктора** | UI для мониторинга сессий в реальном времени и постфактум, вынесения вердиктов |
| **Административная панель** | UI управления пользователями, ролями, интеграциями, настройками |
| **API Gateway** | Аутентификация JWT, rate limiting, маршрутизация запросов к сервисам |
| **Course Service** | Хранение структуры курсов, модулей, учебных материалов, заданий |
| **Assessment Service** | Генерация вопросников, приём ответов, автоматическая и ручная проверка, расчёт баллов |
| **Proctoring Service** | Управление экзаменационными сессиями, детекция тревог, передача видеопотока, хранение вердиктов |
| **Notification Service** | Рассылка email и push-уведомлений по событиям (старт экзамена, результат, нарушение) |
| **Reporting Service** | Формирование отчётов об успеваемости для всех ролей, экспорт CSV/PDF |
| **SIS Sync Service** | Передача итоговых оценок в SIS, retry-механизм, очередь |
| **IAM-модуль** | Управление сессиями после SSO, разграничение доступа по ролям (RBAC) |
| **PostgreSQL** | Транзакционные данные: пользователи, курсы, результаты тестов, события прокторинга, вердикты |
| **Redis** | Кэш сессий, расписания, горячих профилей пользователей, rate limiting счётчики |
| **RabbitMQ** | Брокер асинхронных событий для уведомлений, синхронизации с SIS, передачи вердиктов |

### 3.2 Снаружи SoI

| Внешняя сущность | Данные на входе в SoI | Данные на выходе из SoI | Почему вне границ |
|---|---|---|---|
| **IdP / SSO** | Токен + атрибуты пользователя | Authorization Request | Корпоративная система организации; LMS+P только проверяет токены |
| **WebRTC Медиасервер** | Статус соединения, ID потока | Видеопоток студента, команды записи | Специализированная медиаинфраструктура, масштабируется отдельно |
| **S3-хранилище** | Pre-signed URL | Видеозапись сессии | Объектное хранилище — сторонний сервис с собственным управлением жизненным циклом |
| **SIS / Деканат** | Данные зачисления, учебные планы | Итоговые оценки, статусы | Академическая система организации; LMS+P только обменивается данными по API |
| **SMTP** | Статус доставки | Письма участникам | Инфраструктура почтовой доставки — вне ответственности LMS+P |
| **Античит-агент** | Статус lockdown | Команда активации | Опциональный клиентский агент; разрабатывается/поставляется отдельно |
| **APM / Мониторинг** | Алерты | Метрики, трейсы, логи | Инфраструктурный сервис наблюдаемости; LMS+P только публикует данные |
| **BI / Аналитика** | — | Агрегированные данные (ETL) | Аналитическая платформа организации; LMS+P предоставляет данные для выгрузки |

## 4. Допущения

Следующие допущения приняты при проектировании системы и ограничивают область проектных решений.

| № | Допущение | Последствие нарушения |
|---|---|---|
| A-01 | Организация располагает корпоративным IdP (Keycloak, Azure AD или совместимым), поддерживающим OIDC/SAML 2.0 | Потребуется разработка альтернативного механизма аутентификации |
| A-02 | У студента есть стабильное интернет-соединение ≥ 2 Мбит/с и устройство с работающей веб-камерой при прокторинге | При худших условиях падает качество детекции и вероятность успешного WebRTC |
| A-03 | SIS организации предоставляет REST API для чтения зачислений и записи оценок | При отсутствии API потребуется ручной или файловый импорт/экспорт |
| A-04 | S3-совместимое хранилище (AWS S3, MinIO, Yandex Object Storage) доступно и настроено администратором | Без хранилища невозможен асинхронный прокторинг и хранение записей |
| A-05 | Организация соблюдает требования GDPR / 152-ФЗ и явно получает согласие студентов на видеозапись | Без согласия использование видеопрокторинга юридически недопустимо |
| A-06 | Нагрузка на систему прогнозируема: пиковое число одновременных сессий ≤ 500, пиковое число пользователей ≤ 10 000 | При значительном превышении потребуется пересмотр архитектуры медиасервера |
| A-07 | Плановое обслуживание системы выполняется вне периодов экзаменационных сессий | Нарушение требует blue/green deployment и zero-downtime upgrade |
| A-08 | Роли пользователей (Студент, Преподаватель, Проктор, Администратор) передаются из IdP в атрибутах токена и не управляются вручную внутри LMS+P | Если роли не передаются IdP, требуется дополнительный механизм назначения ролей |
