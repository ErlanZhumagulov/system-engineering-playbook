## 1. Стиль и ключевая идея

**Стиль архитектуры:** Event-driven микросервисная архитектура с CQRS-подходом для разделения операций обучения/оценивания и аналитики.

**Ключевая идея:** Платформа оркестрирует взаимодействие между студентом, медиасервером, движком прокторинга и внешними системами (SIS, S3, SSO) — обеспечивая сохранность данных и справедливость оценивания даже при частичном сбое внешних зависимостей.

**Три логических слоя:**

- **Слой входа и безопасности** — API Gateway: проверка JWT от IdP, rate limiting, маршрутизация. Единая точка входа, все коммуникации только по HTTPS.
- **Ядро обучения и прокторинга (Command-слой)** — Course, Assessment, Proctoring, Notification Service. Все критические события публикуются в RabbitMQ — гарантия сохранности данных при разрыве соединения.
- **Слой управления и аналитики (Query-слой)** — Admin Portal, Reporting Service, SIS Sync Service. Чтение отчётности — из реплики PostgreSQL, чтобы тяжёлые запросы не влияли на приём ответов экзамена.

## 2. Декомпозиция на контейнеры (C4 Level 2)

| № | Контейнер | Тип | Обязанность | Хранит | Технологии |
|---|---|---|---|---|---|
| 1 | **API Gateway** | Сервис | Аутентификация JWT, rate limiting, маршрутизация запросов к downstream-сервисам | stateless | Go / Nginx + Redis |
| 2 | **Course Service** | Сервис | CRUD курсов, модулей, материалов, тестов; управление публикацией; запрос зачисления из SIS | stateless | Go + PostgreSQL |
| 3 | **Assessment Service** | Сервис | Приём ответов студентов, автоматическое оценивание, управление очередью ручной проверки, расчёт итогового балла | stateless | Go + PostgreSQL |
| 4 | **Proctoring Service** | Сервис | Управление экзаменационными сессиями, WebRTC-сигналинг, детекция тревог, управление вердиктами, интеграция с S3 | stateless | Go + WebRTC SDK + PostgreSQL |
| 5 | **Notification Service** | Сервис | Отправка email (SMTP) и push-уведомлений в UI; слушает RabbitMQ-очередь событий | stateless | Go + RabbitMQ + SMTP |
| 6 | **SIS Sync Service** | Сервис | Передача итоговых оценок в SIS; retry-механизм (1/5/15/60 мин); управление очередью `grade_sync_queue` | stateful (sync_queue в PostgreSQL) | Go + PostgreSQL + RabbitMQ |
| 7 | **Reporting Service** | Сервис | Отчёты успеваемости, экспорт CSV/PDF; читает из PostgreSQL-реплики (CQRS Query-сторона) | stateless | Go / Python + PostgreSQL replica |
| 8 | **Admin Portal** | SPA + API | Управление пользователями, ролями, интеграциями; дашборд статусов; аудит-логи | stateless (сессии в Redis) | React + Go API + PostgreSQL |
| 9 | **PostgreSQL** | СУБД | Транзакционные данные: пользователи, курсы, результаты, события прокторинга, вердикты, аудит-логи, sync-очередь | stateful | PostgreSQL 15+ с потоковой репликой + PgBouncer |
| 10 | **Redis** | Кэш / Брокер | Кэш сессий (TTL 8ч), расписания (TTL 24ч), профилей (TTL 5мин); rate limiting счётчики; JWT blacklist | stateful (RAM + RDB snapshot) | Redis 7+ + Sentinel |
| 11 | **RabbitMQ** | Брокер сообщений | Асинхронная доставка событий: `exam.answer.saved`, `exam.session.completed`, `grade.confirmed`, `notification.requested`, `proctoring.alert.detected` | stateful (persistent queues) | RabbitMQ 3.12+ |

## 3. Архитектурные решения (ADR)

---

### ADR-001: WebRTC через выделенный медиасервер (SFU), а не P2P

**Статус:** Принято
**Связанные требования:** SYS-15, SYS-16, SYS-20, NFR-02, NFR-06

**Контекст:**
Прокторинг требует, чтобы проктор мог наблюдать за студентами в реальном времени. Возможны два подхода: P2P WebRTC (браузер студента ↔ браузер проктора) и SFU медиасервер (браузер → сервер → проктор).

**Решение:**
Использовать выделенный медиасервер (SFU — Selective Forwarding Unit, Janus или MediaSoup). Поток студента публикуется на сервер, проктор подписывается на нужные потоки через сервер.

| Критерий | SFU (выбрано) | P2P | RTMP/HLS |
|---|---|---|---|
| Масштабируемость прокторинга (≥20 сессий) | ✅ Да | ❌ Нет | ✅ Да |
| Серверная запись | ✅ Да | ❌ Нет | ✅ Да |
| Задержка | ✅ < 500 мс | ✅ < 200 мс | ❌ 5–30 сек |
| Устойчивость к NAT | ✅ Да | ❌ Проблемы | ✅ Да |
| Операционная стоимость | ⚠️ Выше | ✅ Нулевая | ✅ Ниже |

**Почему выбрано:** Централизованная запись и возможность наблюдения за 20+ сессиями одновременно — критические требования NFR-06 (SR-05). Стоимость медиасервера оправдана. RTMP отклонён из-за недопустимой задержки (NFR-02).

### ADR-002: Двухрежимный прокторинг (realtime + async)

**Статус:** Принято
**Связанные требования:** SYS-14, SYS-16, SYS-20, NFR-06, SR-11

**Контекст:**
Разные типы экзаменов требуют разного уровня контроля. Обязательный прокторинг в реальном времени для всех тестов создаёт высокую нагрузку на проктаров и медиасервер — особенно при массовых тестированиях (>500 студентов).

**Решение:**
Реализовать оба режима прокторинга, выбор на уровне курса/теста.

| Критерий | Только realtime | Только async | Оба режима (выбрано) |
|---|---|---|---|
| Возможность вмешательства проктора | ✅ | ❌ | ✅ |
| Нагрузка на медиасервер | ❌ Высокая | ✅ Нет | ⚠️ Зависит от режима |
| Нагрузка на проктаров | ❌ Высокая | ✅ Гибкая | ✅ Гибкая |
| Сложность реализации | ✅ Один путь | ✅ Один путь | ⚠️ Два пути |

**Почему выбрано:** Массовые промежуточные тесты не требуют наблюдения в реальном времени (SR-11). Два режима снижают операционные затраты без потери контроля над итоговыми экзаменами.

### ADR-003: Event-driven архитектура с RabbitMQ для критических событий

**Статус:** Принято
**Связанные требования:** SYS-09, SYS-24, SYS-25, NFR-08, SR-07

**Контекст:**
Передача ответов студентов, уведомления, синхронизация с SIS — все эти операции должны гарантированно завершаться даже при временных сбоях внешних систем.

**Решение:**
RabbitMQ с persistent queues для всех критических событий: `exam.answer.saved`, `exam.session.completed`, `grade.confirmed`, `notification.requested`. Основной поток синхронный (ответы сохраняются в PostgreSQL), побочные эффекты (уведомления, SIS) — через очередь.

| Критерий | Синхронный REST | RabbitMQ (выбрано) | Kafka |
|---|---|---|---|
| Гарантия доставки | ❌ Нет | ✅ Да | ✅ Да |
| Изоляция сбоев | ❌ Нет | ✅ Да | ✅ Да |
| Сложность эксплуатации | ✅ Простая | ⚠️ Средняя | ❌ Высокая |
| Соответствие нагрузке (200 RPS) | ✅ | ✅ | ⚠️ Избыточно |

**Почему выбрано:** Падение SIS не должно блокировать выставление оценок (SR-07). Kafka избыточна при целевых 200 RPS — RabbitMQ достаточно и проще в эксплуатации.

### ADR-004: CQRS для разделения нагрузки оценивания и отчётности

**Статус:** Принято
**Связанные требования:** SYS-26, SYS-27, NFR-03, NFR-01

**Контекст:**
Тяжёлые отчёты (успеваемость группы, агрегаты по курсу) и транзакционные операции (приём ответов на экзамене, сохранение вердиктов) конкурируют за ресурсы одной БД. В период экзамена запрос отчёта не должен влиять на latency API ответов.

**Решение:**
Записи — в PostgreSQL master (Assessment/Proctoring Service). Чтение для отчётности — из PostgreSQL streaming replica (Reporting Service). Горячие данные (расписание, профили) — в Redis.

| Критерий | Единая БД | CQRS (выбрано) |
|---|---|---|
| Влияние отчётов на транзакционную нагрузку | ❌ Есть | ✅ Нет (read-replica) |
| Независимое масштабирование | ❌ | ✅ |
| Репликационная задержка | — | ⚠️ До 1 сек |
| Сложность | ✅ Простая | ⚠️ Средняя |

**Почему выбрано:** В период пиковой нагрузки (500 экзаменов одновременно) запросы отчётности не должны деградировать API приёма ответов (NFR-01, NFR-03). Задержка репликации до 1 сек приемлема для аналитических отчётов.

### ADR-005: Хранение видеозаписей в S3, а не в БД или файловой системе

**Статус:** Принято
**Связанные требования:** SYS-16, SYS-29, SYS-32, NFR-04, NFR-06

**Контекст:**
Видеозаписи экзаменов могут достигать 1–2 ГБ на сессию. При 500 одновременных экзаменах — сотни гигабайт ежедневно. Хранение в СУБД невозможно, в файловой системе сервиса — не масштабируется и усложняет backup.

**Решение:**
После завершения сессии Proctoring Service загружает запись в S3-совместимое хранилище. Доступ — только через pre-signed URL с TTL 4 часа. Lifecycle policy удаляет объекты по истечении срока хранения.

| Критерий | Файловая система | S3 (выбрано) | СУБД |
|---|---|---|---|
| Масштабируемость | ❌ Ограничена | ✅ Неограничена | ❌ Невозможна |
| Lifecycle / TTL | ❌ Вручную | ✅ Из коробки | ❌ Вручную |
| Шифрование | ⚠️ Вручную | ✅ SSE-S3/KMS | ⚠️ Частично |
| Pre-signed URL | ❌ | ✅ | ❌ |
| Стоимость | ⚠️ Зависит от инфра | ✅ Низкая | ❌ Очень высокая |

**Почему выбрано:** S3 — единственный вариант, удовлетворяющий одновременно масштабируемости (NFR-06), требованиям TTL/приватности (SYS-32, NFR-04) и стоимости при объёмах прокторинга.
