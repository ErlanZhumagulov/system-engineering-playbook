# NFR — Нефункциональные требования

---

Нефункциональные требования определяют качественные характеристики системы. Каждое NFR привязано к измеримым метрикам и методу проверки.

## NFR-01: Доступность

**Описание:**
Система должна быть доступна для проведения экзаменов в соответствии с академическим расписанием. Плановое обслуживание не должно проводиться в период экзаменационных сессий.

**Реализация:**
- Горизонтальное масштабирование API-сервисов (минимум 2 реплики каждого)
- PostgreSQL с потоковой репликацией и автофейловером (Patroni)
- Health checks (`/health`, `/ready`) для всех сервисов
- Blue/green deployment для обновлений без даунтайма

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Uptime в период экзаменационной сессии | ≥ 99.9% (≤ 43 мин/мес) | Prometheus/Grafana, внешний мониторинг |
| Uptime в обычный период | ≥ 99.5% | Prometheus/Grafana |
| Плановое окно обслуживания | ≤ 4 ч/мес, только вне сессий | Регламент обслуживания |
| RTO (Recovery Time Objective) | ≤ 15 минут | Тест восстановления |
| RPO (Recovery Point Objective) | ≤ 5 минут | Тест потери данных |

## NFR-02: Производительность видеопотока

**Описание:**
Система должна обеспечивать стабильную передачу видеопотока прокторинга без заметных задержек и артефактов.

**Реализация:**
- WebRTC с адаптивным битрейтом (REMB, TWCC)
- Медиасервер с горизонтальным масштабированием воркеров
- CDN для раздачи статики и записей

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Задержка видеопотока (glass-to-glass) | ≤ 500 мс | Инструментальное измерение |
| Качество видео | ≥ 720p при стабильном канале | Конфигурация WebRTC |
| Время установления WebRTC-соединения (p95) | ≤ 30 сек | Нагрузочный тест |
| Максимальное число одновременных сессий | ≥ 500 | Нагрузочный тест |
| Процент успешных подключений | ≥ 97% | Мониторинг |

## NFR-03: Производительность API

**Описание:**
Основные операции системы должны отвечать в пределах допустимого времени при расчётной нагрузке.

**Реализация:**
- Redis-кэш для горячих данных (курсы, профили, расписание)
- Индексы в PostgreSQL, connection pooling (PgBouncer)
- Асинхронная обработка через RabbitMQ для уведомлений и передачи в SIS

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Время ответа API (p95, основные операции) | ≤ 500 мс | Нагрузочный тест (k6/JMeter) |
| Время ответа API (p99) | ≤ 2000 мс | Нагрузочный тест |
| Пропускная способность | ≥ 200 RPS | Нагрузочный тест |
| Время загрузки списка курсов | ≤ 1 сек | Функциональный тест |
| Время экспорта отчёта (10 000 записей) | ≤ 10 сек | Функциональный тест |

## NFR-04: Приватность и защита данных

**Описание:**
Система должна соответствовать требованиям GDPR и 152-ФЗ в части обработки персональных данных и видеозаписей.

**Реализация:**
- Видеозаписи доступны только через pre-signed URL с ограниченным TTL
- Аудит-лог всех обращений к записям (неизменяемый)
- Автоматическое удаление записей по истечении срока (S3 lifecycle)
- Право на удаление данных студента в течение 30 дней
- Данные хранятся только в юрисдикции, согласованной с заказчиком

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Доля записей с просроченным TTL pre-signed URL, доступных публично | 0% | Аудит S3 policy |
| Срок выполнения запроса на удаление данных | ≤ 30 дней | Функциональный тест |
| Покрытие аудит-логом доступов к записям | 100% | Инспекция кода + тест |
| Уязвимости класса OWASP Top 10 | 0 критических | Penetration testing |
| Хранение видеозаписей в зашифрованном виде | 100% | Инспекция конфигурации S3 |

## NFR-05: Справедливость прокторинга (качество детекции)

**Описание:**
Автоматическая детекция нарушений должна минимизировать ложноположительные срабатывания, не наказывая добросовестных студентов, и одновременно сохранять достаточную чувствительность к реальным нарушениям.

**Реализация:**
- Confidence score для каждого события — проктор принимает итоговое решение
- Пороговые значения настраиваются администратором
- Регулярный анализ вердиктов для калибровки модели (feedback loop)

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| False Positive Rate (ложные тревоги, вердикт «не нарушение») | ≤ 15% | Анализ вердиктов за период |
| False Negative Rate (пропущенные нарушения) | ≤ 5% | Аудит выборки сессий |
| Доля событий с confidence_score > 0.8 | ≥ 70% от всех тревог | Мониторинг |
| Среднее время вынесения вердикта проктором | ≤ 5 минут на сессию | Инструментальное |

## NFR-06: Масштабируемость

**Описание:**
Система должна поддерживать рост числа одновременных пользователей и экзаменационных сессий без изменения архитектуры.

**Реализация:**
- Stateless API-сервисы — горизонтальное масштабирование через Kubernetes HPA
- Медиасервер — масштабирование по числу активных сессий (K8s KEDA)
- Шардирование БД по `organization_id` при росте >10 000 организаций

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Максимальное число одновременных пользователей | ≥ 10 000 | Расчётное моделирование |
| Максимальное число одновременных экзаменационных сессий | ≥ 500 | Нагрузочный тест |
| Время добавления нового экземпляра API (K8s scale-out) | ≤ 60 сек | Kubernetes тест |
| Линейность прироста производительности | +75% при удвоении ресурсов | Нагрузочный тест |

## NFR-07: Безопасность

**Описание:**
Система должна защищать данные студентов, академические результаты и видеозаписи от несанкционированного доступа.

**Реализация:**
- Все коммуникации только по HTTPS (TLS 1.2+), HSTS
- JWT-токены с коротким TTL (8 ч) и механизмом отзыва (blacklist в Redis)
- Rate limiting на всех публичных API (SYS-33)
- Разграничение данных организаций (multi-tenancy isolation через `organization_id`)

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Критические уязвимости (CVSS ≥ 9.0) | 0 | Penetration testing (OWASP) |
| Высокие уязвимости (CVSS 7.0–8.9) | 0 неустранённых в продакшне | Penetration testing |
| Доля трафика по HTTPS | 100% | Инспекция конфигурации |
| Изоляция данных между организациями | 0 нарушений | Тест на кросс-тенант доступ |
| MTTD несанкционированного доступа | ≤ 2 мин | Мониторинг + алерты |

## NFR-08: Отказоустойчивость и деградация

**Описание:**
При сбое отдельных компонентов или внешних систем система должна сохранять работоспособность в деградированном режиме, не теряя данные.

**Реализация:**
- Circuit breaker для всех вызовов внешних сервисов (SYS-35)
- RabbitMQ с persistent messages для уведомлений и передачи оценок
- Graceful degradation: при недоступности медиасервера — асинхронный режим
- localStorage на клиенте как fallback для несохранённых ответов

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Потеря данных ответов студента при разрыве соединения | 0% | Тест с симуляцией разрыва |
| Время обнаружения сбоя (MTTD) | ≤ 2 мин | Мониторинг (Prometheus) |
| Время восстановления (MTTR) | ≤ 15 мин | Тест восстановления |
| Доля успешных retry для передачи оценок в SIS | ≥ 99% | Анализ логов |

## NFR-09: Наблюдаемость

**Описание:**
Система должна предоставлять достаточно информации для мониторинга, диагностики инцидентов и анализа производительности.

**Реализация:**
- Структурированные JSON-логи (INFO/WARN/ERROR) во всех сервисах
- Prometheus-метрики: RPS, error rate, latency (p50/p95/p99)
- Distributed tracing (OpenTelemetry / Jaeger)
- Grafana-дашборды: технические метрики + бизнес-метрики (активные сессии, события тревоги)

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| MTTD инцидента через алерты | ≤ 2 мин | Мониторинг |
| Срок хранения логов | ≥ 90 дней | Конфигурация |
| Трассируемость запроса от API до БД (критические пути) | 100% | Инспекция |
| Доступность дашбордов Grafana | ≥ 99.9% | Мониторинг |

## NFR-10: Совместимость браузеров и устройств

**Описание:**
Студент должен иметь возможность пройти экзамен с прокторингом в современном браузере без установки дополнительного ПО (кроме опционального античит-агента).

**Реализация:**
- Поддержка WebRTC в Chrome, Firefox, Edge, Safari (последние 2 мажорные версии)
- Адаптивный UI для разрешений от 1024×768
- Graceful degradation при отсутствии WebRTC — сообщение с инструкцией по обновлению браузера

| Метрика | Целевое значение | Метод проверки |
|---|---|---|
| Поддерживаемые браузеры | Chrome, Firefox, Edge, Safari (2 последние мажорные версии) | Тест на матрице браузеров |
| Успешный запуск WebRTC в поддерживаемых браузерах | ≥ 97% | Функциональный тест |
| Минимальное разрешение экрана | 1024×768 | UI-тест |
| Время первой загрузки страницы экзамена (LTE-соединение, 10 Мбит/с) | ≤ 3 сек | Lighthouse / WebPageTest |
