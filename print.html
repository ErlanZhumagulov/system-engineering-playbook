1. Суть кейса и миссия

Система: LMS+P — платформа дистанционного обучения с интегрированным прокторингом.

Цель системы: Обеспечить образовательным организациям надёжный, доступный и справедливый инструмент дистанционного обучения и верифицированной оценки знаний — с защитой от академической нечестности, соблюдением требований приватности и минимальной нагрузкой на административный персонал.

Ключевые задачи LMS+P:

  • Размещение курсов, лекций, заданий и учебных материалов
  • Проведение тестов и экзаменов с автоматической проверкой
  • Видеопрокторинг в реальном времени и в асинхронном режиме
  • Автодетекция нарушений (смена вкладок, посторонние лица, посторонний звук)
  • Передача итоговых оценок в систему деканата (SIS)
  • Отчётность об успеваемости для студентов, преподавателей и администрации
  • Единый вход через SSO организации

2. Контекст — внешние сущности и потоки

LMS+P функционирует в экосистеме образовательной организации. Система выступает как связующий слой между IdP/SSO, системой деканата (SIS), медиасервером, объектным хранилищем и почтовым сервером — объединяя процессы обучения, оценивания, прокторинга и академической отчётности в единый управляемый контур.

2.2 Исполнители

Исполнители — сотрудники образовательной организации, использующие систему как инструмент управления учебным процессом.

РольОсновные действияЧастотаТребования к интерфейсу
ПреподавательСоздание курсов и тестов, просмотр успеваемости, ручная проверка, реакция на нарушенияЕжедневноПолнофункциональный редактор, отчёты
ПрокторМониторинг видеосессий в реальном времени, просмотр записей, вынесение вердиктовВ период экзаменовМинималистичный, быстрый, плитки сессий
АдминистраторУправление учётными записями, ролями, интеграциями, системными настройкамиПо мере необходимостиПанель управления, аудит-логи, статусы

Поведенческий профиль исполнителей:

  • Проктор должен успевать наблюдать за ≥20 сессиями одновременно — интерфейс обязан быть минималистичным, тревоги должны выводиться автоматически на первый план без ручного поиска.
  • Администратор не должен вручную переносить данные между LMS и деканатом — любая синхронизация должна выполняться системой автоматически с уведомлением об ошибках.

2.3 Интеграции с внешними системами

Интеграция №1: IdP / SSO (OIDC / SAML 2.0)

ПараметрОписание
ТипФедеративная аутентификация
ПротоколOIDC (OpenID Connect) или SAML 2.0
Данные от IdP → LMS+Psub (user_id), email, name, roles, groups
Данные от LMS+P → IdPAuthorization Request, redirect_uri
Поведение при недоступностиОтображает страницу ошибки «Сервис авторизации недоступен», активные сессии (JWT) не прерываются до истечения TTL 8 ч
НастройкаАдминистратор указывает Issuer URL, Client ID и Secret в настройках интеграции

Обоснование: LMS+P не хранит пароли и не проводит первичную аутентификацию. Использование корпоративного IdP позволяет применять уже существующую политику паролей и MFA организации, сокращая поверхность атаки и затраты на управление учётными записями. OIDC предпочтителен перед SAML как более современный и гибкий протокол; SAML поддерживается для интеграции с legacy-системами.

Интеграция №2: WebRTC Медиасервер (Janus / MediaSoup)

ПараметрОписание
ТипВидеостриминг реального времени (SFU — Selective Forwarding Unit)
ПротоколWebRTC (DTLS-SRTP) + Signaling через WebSocket
Данные от браузера студента → МедиасерверSDP-оффер, ICE-кандидаты, видео/аудио медиапоток
Данные от Медиасервера → LMS+PСтатус подключения, ID потока, событие о разрыве
Данные от Медиасервера → ПрокторВидеопоток в реальном времени
Поведение при недоступностиСистема переключается в асинхронный режим — студент продолжает экзамен, видео записывается локально

Обоснование: Прямое P2P WebRTC-соединение между студентом и проктором не масштабируется (один проктор не может наблюдать за 20+ студентами через P2P) и не позволяет сохранять запись на сервере. Медиасервер типа SFU решает обе проблемы: проктор подписывается на нужные потоки без нагрузки на устройство студента, а запись формируется централизованно.

Интеграция №3: SIS / Деканат

ПараметрОписание
ТипREST API двусторонний
ПротоколHTTPS, JSON
Данные от SIS → LMS+PСписок зачисленных студентов, учебные планы, расписание экзаменов
Данные от LMS+P → SISИтоговые оценки: student_id, course_id, grade, date, status
СинхронизацияАвтоматически при выставлении итоговой оценки; ручной запуск доступен администратору
Поведение при недоступностиLMS+P сохраняет оценки в очередь и отправляет при восстановлении (retry: 1/5/15/60 мин)

Обоснование: Ручной перенос оценок между LMS и деканатом является источником ошибок и задержек. REST API обеспечивает автоматическую двустороннюю синхронизацию: LMS+P получает актуальный список студентов из SIS при публикации курса, а оценки передаются обратно без участия администратора. Очередь с retry исключает потерю данных при временной недоступности SIS.

Интеграция №4: S3-хранилище

ПараметрОписание
ТипОбъектное хранилище
ПротоколS3 API (совместимый: AWS S3, MinIO, Yandex Object Storage)
Данные от LMS+P → S3Видеозапись сессии (MP4/WebM), метаданные (student_id, exam_id, timestamp)
Данные от S3 → LMS+PPre-signed URL для просмотра записи проктором
Срок храненияНастраивается администратором (по умолчанию 90 дней, S3 lifecycle policy)
ШифрованиеAES-256 на стороне S3 (SSE-S3 или SSE-KMS)

Обоснование: Видеозапись одной экзаменационной сессии составляет 1–2 ГБ. Хранение в реляционной СУБД невозможно, в файловой системе сервиса — не масштабируется и усложняет резервное копирование. S3-совместимое хранилище обеспечивает горизонтальную масштабируемость, встроенное шифрование, автоматическое управление жизненным циклом объектов (TTL-удаление) и возможность выдачи временных pre-signed URL — без раскрытия прямых ссылок на файлы.

2. Сценарии использования

Сценарий 1: Экзамен с прокторингом в реальном времени

Предусловие: Преподаватель создал экзамен с режимом «Реальное время». Проктор назначен на сессию. Студент зачислен на курс и получил уведомление.

Основной поток:

  1. Студент входит в систему через SSO за 15 минут до начала
  2. Открывает страницу экзамена — система проверяет расписание, доступ разрешён
  3. Студент читает политику прокторинга и нажимает «Согласен»; система выполняет camera check
  4. Студент нажимает «Начать экзамен» — Proctoring Service запрашивает WebRTC-соединение с медиасервером
  5. Соединение установлено (≤30 сек); в углу экрана — индикатор «Запись ведётся»
  6. Проктор видит видеопоток студента в портале проктора
  7. Студент отвечает на вопросы; Assessment Service фиксирует ответы с временными метками
  8. Proctoring Service детектирует события: уход из кадра, смена вкладки, посторонний звук
  9. При confidence_score > 0.8 — push-уведомление проктору в течение 5 секунд
  10. По истечении времени система блокирует возможность ответа и автоматически завершает сессию
  11. Запись сохраняется в S3; Assessment Service рассчитывает результат по автоматическим вопросам

Исключение — разрыв видеосоединения:

  1. Медиасервер фиксирует разрыв дольше 30 секунд
  2. LMS+P приостанавливает таймер экзамена и уведомляет студента и проктора
  3. Студент получает инструкцию перезагрузить страницу
  4. При восстановлении соединения (<5 минут) таймер возобновляется, сессия продолжается
  5. Если соединение не восстановлено — проктор принимает решение: продлить сессию или зафиксировать как технический сбой

Режим деградации — медиасервер полностью недоступен:

  1. LMS+P не может установить WebRTC-соединение (таймаут 10 сек)
  2. Система автоматически предлагает студенту переключиться в режим «Асинхронный»
  3. Студент получает уведомление: «Прямой эфир недоступен, используется запись на устройстве»
  4. По завершении экзамена браузерная запись загружается в S3 для последующего просмотра проктором
  5. Проктор уведомляется о переключении режима; сессия помечается proctoring_mode=async_fallback

Сценарий 2: Создание курса и зачисление студентов

Предусловие: Преподаватель авторизован. Курс ещё не создан. SIS содержит актуальные данные о зачислении.

Основной поток:

  1. Преподаватель открывает редактор курса, создаёт модули и загружает материалы (PDF, видео)
  2. Добавляет тест: создаёт вопросы (одиночный выбор, множественный, открытый текст)
  3. Задаёт настройки теста: время, количество попыток, режим прокторинга
  4. Нажимает «Опубликовать» — LMS+P запрашивает из SIS список зачисленных студентов
  5. Студентам предоставляется доступ к курсу; они получают email-уведомление
  6. Преподаватель просматривает прогресс студентов в разделе успеваемости в режиме реального времени

Исключение — ошибка загрузки файла:

  1. Преподаватель загружает файл >2 ГБ или неподдерживаемого формата
  2. Система отображает ошибку: «Максимальный размер файла — 2 ГБ. Поддерживаемые форматы: PDF, MP4, DOCX, PPTX, PNG, JPEG»
  3. Курс сохраняется в черновике; материал не загружается
  4. Преподаватель конвертирует файл и повторяет загрузку

Режим деградации — SIS недоступен при публикации:

  1. LMS+P не может получить список зачисленных студентов из SIS
  2. Система публикует курс, но откладывает автоматическое зачисление в очередь
  3. При восстановлении SIS — зачисление выполняется автоматически
  4. Администратор получает уведомление о задержке зачисления с указанием курса и времени ожидания

Сценарий 3: Асинхронный прокторинг — проверка записи

Предусловие: Студент прошёл экзамен. Запись загружена в S3. Проктор назначен на проверку записей данного экзамена.

Основной поток:

  1. Проктор открывает очередь записей в портале
  2. Система загружает запись с pre-signed URL из S3 (TTL = 4 часа)
  3. Проктор просматривает видео; система отображает временную шкалу с метками тревог
  4. Проктор нажимает на метку тревоги — видео перематывается к событию
  5. Проктор выносит вердикт по каждому событию: «Нарушение» / «Не нарушение» / «Требует эскалации»
  6. При выборе «Нарушение» — открывается форма с полем комментария и выбором типа нарушения
  7. Итоговый вердикт (допустить / не допустить к результатам) сохраняется с proctor_id и timestamp
  8. Преподаватель получает уведомление о вердикте; система обновляет статус результата

Исключение — запись недоступна в S3:

  1. При попытке открыть сессию система обнаруживает отсутствие объекта в S3
  2. Проктор видит статус «Запись недоступна — обратитесь к администратору»
  3. Администратор получает автоматический алерт с session_id и exam_id
  4. Администратор выясняет причину: сбой загрузки после экзамена или технический сбой на стороне студента
  5. Принимается решение: провести повторный экзамен или засчитать результат на основе автодетекции событий

Режим деградации — S3 недоступен во время проверки:

  1. Проктор пытается открыть запись — S3 возвращает ошибку (timeout или 5xx)
  2. Система отображает сообщение: «Хранилище временно недоступно. Повторите через несколько минут»
  3. Событие тревоги и метаданные сессии остаются доступны (хранятся в PostgreSQL)
  4. Проктор может изучить список автодетектированных событий без видео и отложить вынесение вердикта
  5. При восстановлении S3 — проктор возвращается к записи с сохранённым прогрессом просмотра

Сценарий 4: Ручная проверка открытых вопросов

Предусловие: Студент завершил тест с открытыми вопросами. Assessment Service поставил задачи ручной проверки в очередь.

Основной поток:

  1. Преподаватель получает email-уведомление о задачах в очереди
  2. Открывает очередь — видит список ответов (имя студента скрыто, если включена слепая проверка)
  3. Читает ответ, вводит балл в допустимом диапазоне, добавляет комментарий
  4. Сохраняет оценку — Assessment Service пересчитывает итоговый балл
  5. Если все вопросы проверены — статус теста меняется на «Проверено»
  6. Студент получает уведомление и видит итоговый балл с комментарием преподавателя

Исключение — преподаватель вводит балл вне допустимого диапазона:

  1. Преподаватель вводит балл выше максимального (например, 110 из 100)
  2. Система отображает inline-ошибку: «Балл должен быть от 0 до 100»
  3. Форма блокирует сохранение до исправления значения
  4. Преподаватель исправляет балл и сохраняет

Режим деградации — преподаватель не проверяет задачи >N дней:

  1. Через 3 дня бездействия система отправляет напоминание преподавателю
  2. Если через 7 дней очередь не обработана — уведомление администратору с указанием курса и числа непроверенных работ
  3. Администратор может переназначить задачи другому преподавателю
  4. Студент, ожидающий результатов >7 дней, получает уведомление: «Проверка задерживается, ожидайте»

Сценарий 5: Передача итоговых оценок в деканат

Предусловие: Все экзамены курса проведены. Преподаватель проверил открытые вопросы. SIS доступен.

Основной поток:

  1. Преподаватель открывает раздел «Итоговые оценки» курса
  2. Проверяет список: все студенты имеют статус «Оценка выставлена»
  3. Нажимает «Передать в деканат»
  4. SIS Sync Service отправляет POST-запрос в SIS с массивом оценок: student_id, course_id, grade, grade_date, status
  5. SIS возвращает подтверждение; LMS+P фиксирует статус «Передано в деканат» с timestamp
  6. Студенты получают email-уведомление о передаче оценок

Исключение — ошибка валидации в SIS:

  1. SIS возвращает ошибку: student_id не найден или course_id не соответствует учебному плану
  2. LMS+P отображает список проблемных записей с кодом ошибки от SIS
  3. Администратор вручную сверяет данные и исправляет расхождение
  4. Преподаватель повторяет передачу только для проблемных записей

Режим деградации — SIS недоступен:

  1. SIS Sync Service фиксирует HTTP 5xx или timeout — помещает оценки в очередь со статусом «Ожидает передачи»
  2. Система автоматически повторяет попытку с интервалами: 1, 5, 15, 60 минут
  3. Администратор видит в дашборде: «Очередь передачи: N записей, следующая попытка через X мин»
  4. После 5 неудачных попыток — алерт администратору по email
  5. При восстановлении SIS — передача выполняется автоматически, статус обновляется; студенты уведомляются

SR-SYS — System Requirements

Системные требования описывают поведение системы на техническом уровне. Каждое требование атомарно, проверяемо и трассируется к стейкхолдерскому требованию.

1. Аутентификация и управление доступом

SYS-01 (← SR-08) Система должна поддерживать аутентификацию через OIDC/SAML 2.0 с внешним IdP; прямая аутентификация по логину/паролю внутри LMS+P не используется.

SYS-02 (← SR-08) Система должна применять ролевую модель доступа (RBAC) с ролями: Студент, Преподаватель, Проктор, Администратор. Каждая роль имеет явно определённый набор разрешений; пересечение разрешений между ролями возможно только через явное назначение нескольких ролей.

SYS-03 (← SR-10) После успешной аутентификации через SSO система должна создавать сессионный JWT-токен с TTL 8 часов. По истечении токена — перенаправлять на IdP для повторной аутентификации без потери контекста (redirect_uri сохраняется).

SYS-04 (← SR-10) При недоступности IdP система должна возвращать страницу с понятным сообщением об ошибке. Активные сессии с действующим JWT не прерываются. Обход аутентификации не допускается.

2. Управление курсами

SYS-05 (← SR-03) Система должна позволять преподавателю создавать курс с иерархической структурой: Курс → Модули → Уроки → Задания, с возможностью вложенности не менее 3 уровней. Курс должен поддерживать черновик и опубликованное состояние.

SYS-06 (← SR-03) Система должна поддерживать загрузку учебных материалов следующих форматов: PDF, MP4, DOCX, PPTX, PNG, JPEG. Максимальный размер одного файла — 2 ГБ. При превышении лимита — информативное сообщение об ошибке без потери других данных курса.

SYS-07 (← SR-03) Система должна предоставлять конструктор тестов с типами вопросов: одиночный выбор, множественный выбор, текстовый ввод, сопоставление. Минимальное количество вопросов в тесте — 1; порядок вопросов может быть фиксированным или случайным.

SYS-08 (← SR-03, SR-07) При публикации курса система должна автоматически запрашивать из SIS список зачисленных студентов и предоставлять им доступ. При недоступности SIS — зачисление откладывается в очередь и выполняется при восстановлении.

3. Тестирование и оценивание

SYS-09 (← SR-01) Система должна фиксировать ответы студента с временными метками каждый раз при переходе между вопросами. Ответы сохраняются на сервере немедленно; при разрыве соединения прогресс не теряется (localStorage как клиентский кэш, синхронизируется при восстановлении).

SYS-10 (← SR-01) По истечении отведённого времени система должна автоматически завершить сессию и сохранить все текущие ответы студента, включая ответ на текущий открытый вопрос.

SYS-11 (← SR-01, SR-04) Система должна автоматически оценивать вопросы типов «одиночный выбор», «множественный выбор» и «сопоставление» немедленно после завершения теста. Результат отображается студенту сразу.

SYS-12 (← SR-04) Для вопросов типа «текстовый ввод» система должна добавлять задачу в очередь ручной проверки преподавателя с привязкой к student_id, test_id, question_id, submission_timestamp.

SYS-13 (← SR-04) Итоговый балл по тесту должен рассчитываться только после проверки всех вопросов (автоматических и ручных) и сохраняться в БД вместе с completion_timestamp. До завершения всех ручных проверок статус теста — pending_review.

4. Прокторинг

SYS-14 (← SR-01, SR-02, SR-11) Система должна поддерживать три режима прокторинга: none (без прокторинга), async (запись для последующей проверки), realtime (проктор наблюдает онлайн). Режим задаётся преподавателем на уровне теста.

SYS-15 (← SR-01) В режиме realtime система должна устанавливать WebRTC-соединение не позднее чем через 30 секунд после нажатия «Начать экзамен» (p95). При ошибке — отображать диагностическое сообщение и предлагать переключение в режим async.

SYS-16 (← SR-06, SR-11) Система должна записывать видеопоток с веб-камеры студента и поток его экрана в течение всей экзаменационной сессии. Запись должна быть загружена в S3 не позднее 5 минут после завершения сессии.

SYS-17 (← SR-02, SR-05) Proctoring Service должен автоматически детектировать следующие события: уход студента из кадра (>5 сек), обнаружение более одного лица, переключение браузерной вкладки или сворачивание окна (>3 сек), посторонняя речь. Каждое событие фиксируется независимо.

SYS-18 (← SR-02) Каждое событие тревоги должно сохраняться с атрибутами: session_id, event_type, timestamp, confidence_score (0.0–1.0), скриншот фрейма (для событий с лицом, TTL pre-signed URL = 1 ч).

SYS-19 (← SR-06) Проктор должен иметь возможность выносить вердикт по каждому событию: «Нарушение» / «Не нарушение» / «Эскалация». Вердикт фиксируется с proctor_id, verdict_timestamp и опциональным комментарием. Вердикт можно изменить до подтверждения итогового решения.

SYS-20 (← SR-05, SR-06, SR-11) В асинхронном режиме система должна предоставлять проктору интерфейс с видеоплеером, временной шкалой событий тревоги и навигацией по меткам. Нажатие на метку перематывает видео к соответствующему моменту.

Архитектура — LMS+P

Содержание

  1. Стиль и ключевая идея
  2. Декомпозиция на контейнеры (C4 Level 2)
  3. Архитектурные решения (ADR)
  4. Общая согласованность

1. Стиль и ключевая идея

Стиль архитектуры: Event-driven микросервисная архитектура с CQRS-подходом для разделения операций обучения/оценивания и аналитики.

Ключевая идея: Платформа оркестрирует взаимодействие между студентом, медиасервером, движком прокторинга и внешними системами (SIS, S3, SSO) — обеспечивая сохранность данных и справедливость оценивания даже при частичном сбое внешних зависимостей.

Три логических слоя:

  • Слой входа и безопасности — API Gateway: проверка JWT от IdP, rate limiting, маршрутизация. Единая точка входа, все коммуникации только по HTTPS.
  • Ядро обучения и прокторинга (Command-слой) — Course, Assessment, Proctoring, Notification Service. Все критические события публикуются в RabbitMQ — гарантия сохранности данных при разрыве соединения.
  • Слой управления и аналитики (Query-слой) — Admin Portal, Reporting Service, SIS Sync Service. Чтение отчётности — из реплики PostgreSQL, чтобы тяжёлые запросы не влияли на приём ответов экзамена.

3. Архитектурные решения (ADR)

ADR-001: WebRTC через выделенный медиасервер (SFU), а не P2P

Статус: Принято Связанные требования: SYS-15, SYS-16, SYS-20, NFR-02, NFR-06

Контекст: Прокторинг требует, чтобы проктор мог наблюдать за студентами в реальном времени. Возможны два подхода: P2P WebRTC (браузер студента ↔ браузер проктора) и SFU медиасервер (браузер → сервер → проктор).

Решение: Использовать выделенный медиасервер (SFU — Selective Forwarding Unit, Janus или MediaSoup). Поток студента публикуется на сервер, проктор подписывается на нужные потоки через сервер.

КритерийSFU (выбрано)P2PRTMP/HLS
Масштабируемость прокторинга (≥20 сессий)✅ Да❌ Нет✅ Да
Серверная запись✅ Да❌ Нет✅ Да
Задержка✅ < 500 мс✅ < 200 мс❌ 5–30 сек
Устойчивость к NAT✅ Да❌ Проблемы✅ Да
Операционная стоимость⚠️ Выше✅ Нулевая✅ Ниже

Почему выбрано: Централизованная запись и возможность наблюдения за 20+ сессиями одновременно — критические требования NFR-06 (SR-05). Стоимость медиасервера оправдана. RTMP отклонён из-за недопустимой задержки (NFR-02).

ADR-002: Двухрежимный прокторинг (realtime + async)

Статус: Принято Связанные требования: SYS-14, SYS-16, SYS-20, NFR-06, SR-11

Контекст: Разные типы экзаменов требуют разного уровня контроля. Обязательный прокторинг в реальном времени для всех тестов создаёт высокую нагрузку на проктаров и медиасервер — особенно при массовых тестированиях (>500 студентов).

Решение: Реализовать оба режима прокторинга, выбор на уровне курса/теста.

Нагрузка на проктаров❌ Высокая✅ Гибкая✅ Гибкая

Почему выбрано: Массовые промежуточные тесты не требуют наблюдения в реальном времени (SR-11). Два режима снижают операционные затраты без потери контроля над итоговыми экзаменами.

ADR-003: Event-driven архитектура с RabbitMQ для критических событий

Статус: Принято Связанные требования: SYS-09, SYS-24, SYS-25, NFR-08, SR-07

Контекст: Передача ответов студентов, уведомления, синхронизация с SIS — все эти операции должны гарантированно завершаться даже при временных сбоях внешних систем.

Решение: RabbitMQ с persistent queues для всех критических событий: exam.answer.saved, exam.session.completed, grade.confirmed, notification.requested. Основной поток синхронный (ответы сохраняются в PostgreSQL), побочные эффекты (уведомления, SIS) — через очередь.

КритерийСинхронный RESTRabbitMQ (выбрано)Kafka

Почему выбрано: Падение SIS не должно блокировать выставление оценок (SR-07). Kafka избыточна при целевых 200 RPS — RabbitMQ достаточно и проще в эксплуатации.

ADR-004: CQRS для разделения нагрузки оценивания и отчётности

Статус: Принято Связанные требования: SYS-26, SYS-27, NFR-03, NFR-01

Контекст: Тяжёлые отчёты (успеваемость группы, агрегаты по курсу) и транзакционные операции (приём ответов на экзамене, сохранение вердиктов) конкурируют за ресурсы одной БД. В период экзамена запрос отчёта не должен влиять на latency API ответов.

Решение: Записи — в PostgreSQL master (Assessment/Proctoring Service). Чтение для отчётности — из PostgreSQL streaming replica (Reporting Service). Горячие данные (расписание, профили) — в Redis.

КритерийЕдиная БДCQRS (выбрано)
Влияние отчётов на транзакционную нагрузку❌ Есть✅ Нет (read-replica)
Независимое масштабирование❌✅
Репликационная задержка—⚠️ До 1 сек
Сложность✅ Простая⚠️ Средняя

Почему выбрано: В период пиковой нагрузки (500 экзаменов одновременно) запросы отчётности не должны деградировать API приёма ответов (NFR-01, NFR-03). Задержка репликации до 1 сек приемлема для аналитических отчётов.

ADR-005: Хранение видеозаписей в S3, а не в БД или файловой системе

Статус: Принято Связанные требования: SYS-16, SYS-29, SYS-32, NFR-04, NFR-06

Контекст: Видеозаписи экзаменов могут достигать 1–2 ГБ на сессию. При 500 одновременных экзаменах — сотни гигабайт ежедневно. Хранение в СУБД невозможно, в файловой системе сервиса — не масштабируется и усложняет backup.

Решение: После завершения сессии Proctoring Service загружает запись в S3-совместимое хранилище. Доступ — только через pre-signed URL с TTL 4 часа. Lifecycle policy удаляет объекты по истечении срока хранения.

КритерийФайловая системаS3 (выбрано)СУБД

Lifecycle / TTL❌ Вручную✅ Из коробки❌ Вручную
Шифрование⚠️ Вручную✅ SSE-S3/KMS⚠️ Частично
Pre-signed URL❌✅❌
Стоимость⚠️ Зависит от инфра✅ Низкая❌ Очень высокая

Почему выбрано: S3 — единственный вариант, удовлетворяющий одновременно масштабируемости (NFR-06), требованиям TTL/приватности (SYS-32, NFR-04) и стоимости при объёмах прокторинга.
