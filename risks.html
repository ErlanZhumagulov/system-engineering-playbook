Детальный реестр рисков

R-01. Отказ медиасервера во время экзамена

ПолеОписание
ПричинаТехнический сбой медиасервера, перегрузка при пиковой нагрузке, сетевой разрыв между LMS+P и медиасервером
СобытиеWebRTC-соединение разрывается; студент теряет возможность передавать видеопоток
ПоследствиеЭкзамен прерывается; прокторинг в реальном времени невозможен; студент несёт психологическую нагрузку
ВероятностьСредняя (2)
УщербВысокий (3) — прямое влияние на ход академического экзамена
Приоритет6
МерыАвтопереключение в асинхронный режим (локальная запись в браузере); таймер сессии паузируется при разрыве >30 сек; горизонтальное масштабирование медиасервера (K8s KEDA); circuit breaker в Proctoring Service
Остаточный рискСредний — асинхронный режим не обеспечивает наблюдения в реальном времени

R-02. Высокий уровень ложноположительных срабатываний детектора нарушений

ПолеОписание
ПричинаПлохое освещение, очки, маска, движение вентилятора, фоновый шум детектируются как нарушение
СобытиеСтуденты без нарушений получают тревоги; проктор перегружен ложными событиями
ПоследствиеНесправедливые вердикты; жалобы студентов; потеря доверия к системе
ВероятностьВысокая (3) — особенно при разнородных домашних условиях
УщербВысокий (3) — репутационный и академический ущерб
Приоритет9
МерыConfidence score + настраиваемый порог; обязательная проверка проктором перед финальным вердиктом; предэкзаменационный camera check; feedback loop для калибровки модели; мониторинг метрики FPR (NFR-05)
Остаточный рискНизкий — FPR ≤ 15% как измеримое NFR; проктор принимает финальное решение

R-03. Недоступность SSO/IdP при проведении экзаменов

ПолеОписание
ПричинаТехнический сбой корпоративного IdP (Keycloak, Azure AD)
СобытиеСтуденты не могут войти в систему; новые сессии не создаются
ПоследствиеЭкзамены срываются; требуется перенос; организационный и репутационный ущерб
ВероятностьНизкая (1) — IdP — критическая инфраструктура организации
УщербКритический (3)
Приоритет3
МерыJWT с TTL 8 ч — активные сессии не прерываются; мониторинг доступности IdP с алертом заблаговременно; инструкция аварийного входа для администратора; рекомендация: IdP с HA-конфигурацией
Остаточный рискНизкий — активные сессии продолжаются; блокируются только новые входы

R-04. Потеря ответов студента при разрыве соединения

ПолеОписание
ПричинаНестабильный интернет у студента, сбой браузера, разряд устройства
СобытиеСтудент теряет связь; несохранённые ответы могут быть утеряны
ПоследствиеЧастичная потеря ответов → несправедливый результат; жалобы и апелляции
ВероятностьСредняя (2)
УщербВысокий (3)
Приоритет6
МерыАвтосохранение на сервере при каждом переходе между вопросами (SYS-09); Idempotency-Key для предотвращения дублей; localStorage как кэш ответов на стороне клиента; синхронизация при восстановлении соединения
Остаточный рискОчень низкий — потеря возможна только для текущего вопроса, если переход ещё не произошёл

R-05. Несанкционированный доступ к видеозаписям экзаменов

ПолеОписание
ПричинаКомпрометация API-ключа S3; утечка pre-signed URL; ошибка в политиках доступа S3
СобытиеПосторонние лица получают доступ к видеозаписям без соответствующих прав
ПоследствиеНарушение приватности студентов; ответственность по GDPR/152-ФЗ; штрафы; репутационный ущерб
ВероятностьНизкая (1)
УщербКритический (3)
Приоритет3
МерыPre-signed URL с TTL 4 ч (SYS-29); аудит-лог всех обращений (SYS-31); ротация S3-ключей; шифрование AES-256 (SYS-29); регулярный аудит политик доступа S3; rate limiting на endpoint получения URL
Остаточный рискНизкий

R-06. Сбой передачи оценок в SIS

ПолеОписание
ПричинаПлановые работы на SIS; изменение API SIS без уведомления; сетевые проблемы
СобытиеОценки из LMS+P не поступают в деканат
ПоследствиеЗадержка в академической документации; студенты не видят официальных оценок в деканате
ВероятностьСредняя (2)
УщербСредний (2) — оценки не теряются, только задерживаются
Приоритет4
МерыОчередь с retry (1/5/15/60 мин) (SYS-25); алерт администратору после 5 неудач; ручная повторная отправка из UI; мониторинг статуса интеграции на дашборде
Остаточный рискНизкий — данные не теряются, доставка задерживается

R-07. Академическое мошенничество через удалённое управление устройством

ПолеОписание
ПричинаСтудент использует TeamViewer, AnyDesk или аналоги во время экзамена
СобытиеТретья сторона управляет компьютером студента без обнаружения
ПоследствиеНечестный результат экзамена; несправедливость по отношению к другим студентам
ВероятностьСредняя (2)
УщербВысокий (3)
Приоритет6
МерыОпциональный античит-агент (lockdown browser) блокирует сторонние приложения; детекция смены активного окна (SYS-17); анализ поведенческих паттернов (скорость ввода — опционально)
Остаточный рискСредний — без lockdown browser полная защита невозможна

R-08. Перегрузка системы в начале экзаменационной сессии

ПолеОписание
ПричинаВсе студенты одновременно входят и запускают сессии в первые 5 минут экзамена
СобытиеРезкий spike нагрузки; API Gateway, Proctoring Service и медиасервер перегружены
ПоследствиеЗадержки и отказы в запуске сессий; студенты теряют время экзамена; жалобы и апелляции
ВероятностьВысокая (3) — предсказуемый паттерн нагрузки
УщербВысокий (3)
Приоритет9
МерыK8s HPA — автомасштабирование по CPU/RPS (NFR-06); предварительное прогревание пула воркеров медиасервера; rate limiting (SYS-33); нагрузочное тестирование перед каждой сессией; рекомендация: разнесение окон начала экзаменов по 5–10 мин
Остаточный рискНизкий — при правильно настроенном HPA и прогреве воркеров

R-09. Несоответствие требованиям GDPR при хранении видеозаписей

ПолеОписание
ПричинаВидеозаписи хранятся дольше установленного срока или используются без согласия студента
СобытиеНарушение политики хранения персональных данных
ПоследствиеШтрафы по GDPR (до 4% годового оборота); судебные иски студентов; репутационный ущерб
ВероятностьНизкая (1)
УщербКритический (3)
Приоритет3
МерыАвтоматическое удаление по TTL (S3 lifecycle policy, SYS-32); согласие студента перед экзаменом (SYS-02); аудит удалений; право на удаление данных (SYS-30); хранение в согласованной юрисдикции
Остаточный рискОчень низкий

R-10. Деградация качества детекции при низком качестве камеры студента

ПолеОписание
ПричинаСтудент использует веб-камеру с низким разрешением или находится при плохом освещении
СобытиеProctoring Service не может корректно обнаружить лицо или события нарушений
ПоследствиеСнижение достоверности прокторинга; повышение FPR или пропуск нарушений
ВероятностьСредняя (2)
УщербСредний (2)
Приоритет4
МерыПредэкзаменационный camera check с оценкой качества изображения; минимальные требования к оборудованию в интерфейсе; fallback: рекомендация улучшить освещение; возможность fallback на асинхронный режим
Остаточный рискНизкий
