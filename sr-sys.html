SR-SYS — System Requirements

Системные требования описывают поведение системы на техническом уровне. Каждое требование атомарно, проверяемо и трассируется к стейкхолдерскому требованию.

1. Аутентификация и управление доступом

SYS-01 (← SR-08) Система должна поддерживать аутентификацию через OIDC/SAML 2.0 с внешним IdP; прямая аутентификация по логину/паролю внутри LMS+P не используется.

SYS-02 (← SR-08) Система должна применять ролевую модель доступа (RBAC) с ролями: Студент, Преподаватель, Проктор, Администратор. Каждая роль имеет явно определённый набор разрешений; пересечение разрешений между ролями возможно только через явное назначение нескольких ролей.

SYS-03 (← SR-10) После успешной аутентификации через SSO система должна создавать сессионный JWT-токен с TTL 8 часов. По истечении токена — перенаправлять на IdP для повторной аутентификации без потери контекста (redirect_uri сохраняется).

SYS-04 (← SR-10) При недоступности IdP система должна возвращать страницу с понятным сообщением об ошибке. Активные сессии с действующим JWT не прерываются. Обход аутентификации не допускается.

2. Управление курсами

SYS-05 (← SR-03) Система должна позволять преподавателю создавать курс с иерархической структурой: Курс → Модули → Уроки → Задания, с возможностью вложенности не менее 3 уровней. Курс должен поддерживать черновик и опубликованное состояние.

SYS-06 (← SR-03) Система должна поддерживать загрузку учебных материалов следующих форматов: PDF, MP4, DOCX, PPTX, PNG, JPEG. Максимальный размер одного файла — 2 ГБ. При превышении лимита — информативное сообщение об ошибке без потери других данных курса.

SYS-07 (← SR-03) Система должна предоставлять конструктор тестов с типами вопросов: одиночный выбор, множественный выбор, текстовый ввод, сопоставление. Минимальное количество вопросов в тесте — 1; порядок вопросов может быть фиксированным или случайным.

SYS-08 (← SR-03, SR-07) При публикации курса система должна автоматически запрашивать из SIS список зачисленных студентов и предоставлять им доступ. При недоступности SIS — зачисление откладывается в очередь и выполняется при восстановлении.

3. Тестирование и оценивание

SYS-09 (← SR-01) Система должна фиксировать ответы студента с временными метками каждый раз при переходе между вопросами. Ответы сохраняются на сервере немедленно; при разрыве соединения прогресс не теряется (localStorage как клиентский кэш, синхронизируется при восстановлении).

SYS-10 (← SR-01) По истечении отведённого времени система должна автоматически завершить сессию и сохранить все текущие ответы студента, включая ответ на текущий открытый вопрос.

SYS-11 (← SR-01, SR-04) Система должна автоматически оценивать вопросы типов «одиночный выбор», «множественный выбор» и «сопоставление» немедленно после завершения теста. Результат отображается студенту сразу.

SYS-12 (← SR-04) Для вопросов типа «текстовый ввод» система должна добавлять задачу в очередь ручной проверки преподавателя с привязкой к student_id, test_id, question_id, submission_timestamp.

SYS-13 (← SR-04) Итоговый балл по тесту должен рассчитываться только после проверки всех вопросов (автоматических и ручных) и сохраняться в БД вместе с completion_timestamp. До завершения всех ручных проверок статус теста — pending_review.

4. Прокторинг

SYS-14 (← SR-01, SR-02, SR-11) Система должна поддерживать три режима прокторинга: none (без прокторинга), async (запись для последующей проверки), realtime (проктор наблюдает онлайн). Режим задаётся преподавателем на уровне теста.

SYS-15 (← SR-01) В режиме realtime система должна устанавливать WebRTC-соединение не позднее чем через 30 секунд после нажатия «Начать экзамен» (p95). При ошибке — отображать диагностическое сообщение и предлагать переключение в режим async.

SYS-16 (← SR-06, SR-11) Система должна записывать видеопоток с веб-камеры студента и поток его экрана в течение всей экзаменационной сессии. Запись должна быть загружена в S3 не позднее 5 минут после завершения сессии.

SYS-17 (← SR-02, SR-05) Proctoring Service должен автоматически детектировать следующие события: уход студента из кадра (>5 сек), обнаружение более одного лица, переключение браузерной вкладки или сворачивание окна (>3 сек), посторонняя речь. Каждое событие фиксируется независимо.

SYS-18 (← SR-02) Каждое событие тревоги должно сохраняться с атрибутами: session_id, event_type, timestamp, confidence_score (0.0–1.0), скриншот фрейма (для событий с лицом, TTL pre-signed URL = 1 ч).

SYS-19 (← SR-06) Проктор должен иметь возможность выносить вердикт по каждому событию: «Нарушение» / «Не нарушение» / «Эскалация». Вердикт фиксируется с proctor_id, verdict_timestamp и опциональным комментарием. Вердикт можно изменить до подтверждения итогового решения.

SYS-20 (← SR-05, SR-06, SR-11) В асинхронном режиме система должна предоставлять проктору интерфейс с видеоплеером, временной шкалой событий тревоги и навигацией по меткам. Нажатие на метку перематывает видео к соответствующему моменту.

5. Расписание и уведомления

SYS-21 (← SR-01, SR-10) Система должна открывать доступ к экзамену строго в соответствии с расписанием (start_time — end_time): блокировать начало до start_time (HTTP 423), автоматически завершать активные сессии по end_time.

SYS-22 (← SR-01) Система должна отправлять студенту email-уведомление за 24 часа и за 1 час до начала экзамена. Уведомление должно содержать название экзамена, временное окно и ссылку на платформу.

SYS-23 (← SR-05, SR-12) При возникновении события тревоги с confidence_score > 0.8 в режиме realtime система должна доставить push-уведомление проктору в интерфейсе в течение 5 секунд от момента детекции.

6. Интеграция с SIS и отчётность

SYS-24 (← SR-07) После подтверждения итоговых оценок преподавателем система должна передавать их в SIS через REST API с полями: student_id, course_id, grade, grade_scale, grade_date, status.

SYS-25 (← SR-07) При ошибке передачи в SIS (HTTP 4xx/5xx или timeout) система должна сохранять оценки в очередь и повторять попытку с интервалами 1, 5, 15, 60 минут. После 5 неудачных попыток — email-уведомление администратору.

SYS-26 (← SR-04) Система должна предоставлять преподавателю отчёт по курсу со следующими метриками: средний балл, распределение оценок (гистограмма), процент успешно сдавших, прогресс по модулям. Данные обновляются в реальном времени.

SYS-27 (← SR-13) Система должна поддерживать экспорт отчётов в форматах CSV и PDF. Время формирования отчёта до 10 000 записей — не более 10 секунд.

7. Безопасность и приватность

SYS-28 (← SR-09, SR-10) Все API-эндпоинты должны быть доступны только по HTTPS с TLS 1.2 и выше. HTTP-запросы перенаправляются на HTTPS (301). HSTS включён для всех доменов.

SYS-29 (← SR-06, SR-09) Видеозаписи экзаменов должны храниться в S3 с шифрованием AES-256 (SSE-S3 или SSE-KMS) и быть доступны только через pre-signed URL с TTL не более 4 часов.

SYS-30 (← SR-09) Система должна обеспечивать возможность удаления всех персональных данных студента (GDPR Right to Erasure) по запросу администратора. Операция должна завершаться в течение 30 дней с момента запроса и включать: профиль, ответы, видеозаписи, аудит-логи (за исключением данных, требуемых для юридической отчётности).

SYS-31 (← SR-09) Система должна вести аудит-лог всех обращений к видеозаписям с полями: user_id, role, session_id, access_timestamp, action (view / download / delete). Аудит-лог неизменяем.

SYS-32 (← SR-09) Видеозаписи должны автоматически удаляться из S3 по истечении настраиваемого срока хранения (по умолчанию 90 дней, минимум 30 дней, максимум 365 дней). Удаление выполняется через S3 lifecycle policy.

8. Дополнительные системные требования

SYS-33 (← SR-10) Система должна применять rate limiting на все публичные API-эндпоинты. Лимит для неаутентифицированных запросов — 20 запросов/мин с одного IP. Лимит для аутентифицированных — 300 запросов/мин с одного user_id.

SYS-34 (← SR-08) Администратор должен иметь возможность настраивать интеграции (IdP, SIS, S3, SMTP) через веб-интерфейс без редактирования конфигурационных файлов или перезапуска сервисов.

SYS-35 (← SR-10) Система должна реализовывать circuit breaker для всех вызовов внешних сервисов (IdP, SIS, SMTP, S3, WebRTC Медиасервер). При срабатывании — fallback-поведение согласно CONOPS, уведомление администратору.
